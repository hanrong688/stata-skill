# Splink v2 Architecture Specification

Full-fidelity probabilistic record linkage for Stata, modeled after
[splink](https://github.com/moj-analytical-services/splink).

## 1. New .ado Syntax

### 1.1 Basic Syntax (Backward-Compatible)

The existing syntax continues to work unchanged:

```stata
splink varlist [if] [in], BLOCKvar(varlist) GENerate(name)
    [THReshold(real 0.85) MAXIter(integer 25) LINKvar(varname) REPlace Verbose]
```

When no `compare()` option is specified, every variable in `varlist` gets the
legacy behavior: 3-level Jaro-Winkler for strings, 3-level distance for
numerics, with hardcoded thresholds (JW 0.92/0.80, numeric tolerance 1.0).

### 1.2 Extended Syntax

```stata
splink varlist [if] [in], BLOCKvar(varlist) GENerate(name)
    [COMPare(comp_spec [; comp_spec ...])
     BLOCKrules(rule_spec [; rule_spec ...])
     THReshold(real 0.85)
     MAXIter(integer 25)
     PRIor(real 0.01)
     LINKvar(varname)
     DEDUPE
     PAIRwise(name)
     FIXm(fix_spec)
     FIXu(fix_spec)
     NULLweight(string)
     TFadjust(varlist)
     REPlace
     Verbose]
```

### 1.3 Option Descriptions

#### compare(comp_spec [; comp_spec ...])

Each `comp_spec` defines how one variable is compared. Semicolons separate
variable-level specifications. Within each spec:

```
varname, method[(threshold_args)] [, levels(int)] [, null(neutral|penalize)]
```

**method** is one of:
- `jw` or `jaro_winkler` -- Jaro-Winkler similarity (default for strings)
- `lev` or `levenshtein` -- Levenshtein edit distance
- `dlev` or `damerau_levenshtein` -- Damerau-Levenshtein distance
- `jaccard` -- Jaccard similarity on character bigrams
- `exact` -- binary exact match (2 levels only: match/non-match)
- `numeric` or `distance` -- absolute numeric distance (default for numerics)

**threshold_args** are method-specific thresholds that define level boundaries,
from most similar to least similar. The number of thresholds determines the
number of non-null levels (n_thresholds + 1 = n_levels, since "below all
thresholds" is the lowest level).

**Examples:**

```stata
* 2 thresholds -> 3 levels (exact, similar, different)
compare(first_name, jw(0.92, 0.80))

* 3 thresholds -> 4 levels
compare(last_name, jw(0.95, 0.88, 0.70))

* Exact match only (2 levels: match or non-match)
compare(gender, exact)

* Numeric with custom distance thresholds
compare(dob_year, distance(0, 1, 5))
*   level 0: distance == 0 (exact)
*   level 1: distance <= 1
*   level 2: distance <= 5
*   level 3: distance > 5

* Levenshtein with single threshold
compare(city, lev(2))
*   level 0: distance <= 2
*   level 1: distance > 2

* Multiple variables
compare(first_name, jw(0.92, 0.80) ; last_name, jw(0.92, 0.80) ; ///
        dob_year, distance(0, 1) ; gender, exact)
```

**Default behavior** (no `compare()` option): each variable in `varlist` gets
the v1 defaults -- `jw(0.92, 0.80)` for strings, `distance(0, 1.0)` for
numerics, with null penalized.

#### blockrules(rule_spec [; rule_spec ...])

Multiple blocking rules with OR logic. Each rule specifies variables that must
match exactly. Semicolons separate rules. Pairs generated by any rule are
deduplicated (a pair appears once even if it matches multiple rules).

```stata
* Single rule (same as blockvar)
blockrules(last_name)

* Two rules: block on last_name OR block on (city, dob_year)
blockrules(last_name ; city dob_year)

* Three rules
blockrules(last_name ; city dob_year ; first_name city)
```

When `blockrules()` is specified, `blockvar()` is ignored (with a warning if
both are present). When neither is specified, error.

**Backward compatibility:** `blockvar(varlist)` remains as a single-rule
shorthand. `blockvar(state dob_year)` is equivalent to
`blockrules(state dob_year)`.

#### prior(real)

Initial match probability (lambda) for EM. Default: 0.01. Acceptable range:
(0, 1). This replaces the hardcoded `LAMBDA_INIT = 0.01`.

#### dedupe

When specified, the command runs in deduplication mode within a single dataset.
This is the default behavior (same as v1). Included for explicit clarity when
used alongside `linkvar()` in documentation.

#### pairwise(name)

Output a variable containing match weights for every candidate pair. When
specified, the command creates a pairwise results dataset instead of (or in
addition to) the cluster assignment.

The pairwise output is written to a tempfile and merged back. It creates
variables:
- `_splink_id_a` -- observation index of record A
- `_splink_id_b` -- observation index of record B
- `_splink_match_weight` -- log2 Bayes factor (sum of partial match weights)
- `_splink_match_prob` -- posterior match probability

These are stored in a frame (Stata 16+) or a separate dataset file whose path
is stored in `r(pairwise_file)`.

#### fixm(fix_spec) and fixu(fix_spec)

Fix m or u probabilities for specific variables instead of estimating them via
EM. The spec is a semicolon-separated list:

```stata
fixm(varname, level0_prob level1_prob level2_prob [; varname2, ...])
```

The number of probabilities must match the number of levels for that variable.
They must sum to 1.0 (within tolerance).

```stata
* Fix m probabilities for first_name (3 levels)
fixm(first_name, 0.95 0.04 0.01)

* Fix u probabilities for gender (2 levels, exact match)
fixu(gender, 0.05 0.95)
```

Variables not listed in `fixm()`/`fixu()` are estimated by EM as usual.

#### nullweight(string)

How to handle null (missing) comparisons. Applies globally unless overridden
per-variable in `compare()`.

- `neutral` -- null comparisons contribute 0 to the match weight (Bayes factor
  = 1). This is the splink default.
- `penalize` -- null comparisons are treated as the lowest level (current v1
  behavior). This remains the default for backward compatibility.

#### tfadjust(varlist)

Enable term frequency adjustment for the listed variables. TF adjustment
modifies the u-probability for exact matches based on how common the matched
value is. Rare values that match are stronger evidence of a true match than
common values that match.

```stata
tfadjust(first_name last_name city)
```

The .ado wrapper computes frequency tables for each TF-adjusted variable and
writes them to the config file. The plugin reads these tables and adjusts
u-probabilities at scoring time.

### 1.4 Complete Example

```stata
* Simple (backward-compatible)
splink first_name last_name dob_year city, block(last_name) gen(cluster_id)

* Full-featured
splink first_name last_name dob_year gender city, ///
    compare(first_name, jw(0.92, 0.80) ; ///
            last_name, jw(0.95, 0.88, 0.70) ; ///
            dob_year, distance(0, 1, 5) ; ///
            gender, exact ; ///
            city, jw(0.92, 0.80)) ///
    blockrules(last_name ; city dob_year) ///
    tfadjust(first_name last_name) ///
    prior(0.05) ///
    threshold(0.90) ///
    maxiter(50) ///
    pairwise(pw_results) ///
    nullweight(neutral) ///
    gen(cluster_id) ///
    replace verbose
```


## 2. Configuration File Format

The .ado wrapper writes a tempfile that the C plugin reads via `fopen()`. This
replaces the limited `argv[]` for complex configuration. The format is
line-oriented plain text, designed for simple `fgets()`/`sscanf()` parsing in C.

### 2.1 File Structure

```
SPLINK_CONFIG_V2
n_comp=<int>
has_link=<0|1>
threshold=<double>
max_iter=<int>
prior=<double>
null_mode=<neutral|penalize>
verbose=<0|1>
pairwise=<0|1>
n_blocking_rules=<int>

[BLOCKING]
rule=<n_vars> <var_indices...>
rule=<n_vars> <var_indices...>
...

[COMPARISONS]
var=<index> type=<string|numeric> method=<jw|lev|dlev|jaccard|exact|distance> n_levels=<int> thresholds=<t1 t2 ...> null=<neutral|penalize> tf=<0|1>
var=<index> ...
...

[FIXED_M]
var=<index> probs=<p0 p1 p2 ...>
...

[FIXED_U]
var=<index> probs=<p0 p1 p2 ...>
...

[TF_TABLE:<var_index>]
<value>\t<frequency>
<value>\t<frequency>
...

[TF_TABLE:<var_index>]
...

[END]
```

### 2.2 Section Details

#### Header

```
SPLINK_CONFIG_V2
n_comp=5
has_link=0
threshold=0.90
max_iter=50
prior=0.05
null_mode=neutral
verbose=1
pairwise=1
n_blocking_rules=2
```

The magic line `SPLINK_CONFIG_V2` identifies the file format. If this line is
absent, the plugin falls back to legacy `argv[]` parsing for backward
compatibility.

#### [BLOCKING]

Each `rule=` line specifies one blocking rule. The first number is the count of
variables in the rule, followed by 1-indexed Stata variable positions (relative
to the plugin call's variable list).

```
[BLOCKING]
rule=1 2
rule=2 5 4
```

This means: rule 1 blocks on variable 2 (last_name), rule 2 blocks on
variables 5 (city) AND 4 (dob_year). A pair matches a rule if it agrees on ALL
variables in that rule. The plugin generates candidate pairs from any rule (OR
logic) and deduplicates.

**Important:** Variable indices here refer to positions in the comparison
variable list (1-indexed within the comp vars, not the full Stata dataset).
The .ado wrapper maps Stata variable names to these indices.

#### [COMPARISONS]

One line per comparison variable. Fields are space-separated key=value pairs.

```
[COMPARISONS]
var=1 type=string method=jw n_levels=3 thresholds=0.92,0.80 null=neutral tf=1
var=2 type=string method=jw n_levels=4 thresholds=0.95,0.88,0.70 null=neutral tf=1
var=3 type=numeric method=distance n_levels=4 thresholds=0,1,5 null=neutral tf=0
var=4 type=string method=exact n_levels=2 thresholds= null=penalize tf=0
var=5 type=string method=jw n_levels=3 thresholds=0.92,0.80 null=neutral tf=0
```

- `var=` -- 1-indexed position in the comparison variable list
- `type=` -- `string` or `numeric`
- `method=` -- comparison function name
- `n_levels=` -- total number of comparison levels (including the "else" level)
- `thresholds=` -- comma-separated threshold values, high to low. For
  `n_levels=K`, there are `K-1` thresholds. For `exact` (2 levels), this is
  empty.
- `null=` -- `neutral` or `penalize` (per-variable override of global setting)
- `tf=` -- `1` if term frequency adjustment is enabled for this variable

#### [FIXED_M] and [FIXED_U]

Optional sections. Only variables with user-fixed probabilities appear here.

```
[FIXED_M]
var=1 probs=0.95,0.04,0.01

[FIXED_U]
var=4 probs=0.05,0.95
```

Probabilities are comma-separated, one per level, must sum to 1.0.

#### [TF_TABLE:<var_index>]

One section per TF-adjusted variable. Contains tab-separated value/frequency
pairs. The frequency is the proportion (0-1) of records with that value.

```
[TF_TABLE:1]
john	0.045
james	0.038
robert	0.032
mary	0.031
...

[TF_TABLE:2]
smith	0.008
johnson	0.006
williams	0.005
...
```

The .ado computes these from `tab varname` before calling the plugin. Values are
lowercased and trimmed to match plugin preprocessing.

### 2.3 Legacy Fallback

When no config file path is provided (or `argv[0]` is not a file path starting
with `/` or containing `.tmp`), the plugin uses the legacy `argv[]` interface:

```
argv[0] = n_comp
argv[1] = is_string flags
argv[2] = has_link
argv[3] = threshold
argv[4] = max_iter
argv[5] = verbose
argv[6] = diag_path
```

This preserves complete backward compatibility with existing compiled plugins.


## 3. C Plugin Interface Changes

### 3.1 argv[] Changes

The new interface uses exactly two arguments:

```
argv[0] = config_path    (path to the config tempfile)
argv[1] = diag_path      (path to write diagnostics)
```

The plugin detects the interface version by checking whether `argv[0]` parses
as an integer (legacy) or as a file path (v2).

```c
STDLL stata_call(int argc, char *argv[]) {
    if (argc >= 7 && isdigit(argv[0][0])) {
        return stata_call_v1(argc, argv);  /* legacy path */
    }
    return stata_call_v2(argv[0], argv[1]); /* new path */
}
```

### 3.2 Variable Layout (Plugin Call)

The .ado wrapper passes variables to the plugin in this order:

```
Var 1..n_comp:           comparison variables
Var n_comp+1:            link_source (if has_link), else absent
Var n_comp+has_link+1:   cluster_id output variable
Var n_comp+has_link+2:   _splink_id_a (if pairwise), else absent
Var n_comp+has_link+3:   _splink_id_b (if pairwise), else absent
Var n_comp+has_link+4:   _splink_match_weight (if pairwise), else absent
Var n_comp+has_link+5:   _splink_match_prob (if pairwise), else absent
```

Note: blocking variables are NOT passed to the plugin. The .ado constructs
blocking keys and passes them via the config file's blocking rules. The plugin
reads the comparison variables and uses them both for blocking (exact match on
specified indices) and comparison.

**Wait -- this does not work.** The plugin needs the raw values to evaluate
blocking rules (exact match on arbitrary subsets of comparison variables). Since
the comparison variables are already passed, blocking rules can reference them
by index. The plugin computes blocking keys internally from the comparison
variable values.

Actually, blocking variables might NOT be comparison variables. In v1,
`blockvar()` and `varlist` are independent. A user might block on `zipcode` but
not compare it. We need to handle this.

**Revised variable layout:**

```
Var 1..n_block_only:                    blocking-only variables (not in varlist)
Var n_block_only+1..n_block_only+n_comp: comparison variables
Var next:                                link_source (if has_link)
Var next:                                cluster_id (output)
[Var next..next+3:                       pairwise output vars (if pairwise)]
```

The config file maps blocking rule variable indices into this combined list.
Comparison variable indices in [COMPARISONS] are offset by `n_block_only`.

This is getting complicated. Let me simplify.

**Final variable layout:**

```
Var 1..n_allvars:     all unique variables (blocking-only + comparison)
Var n_allvars+1:      link_source (if has_link)
Var n_allvars+2:      cluster_id (output)
```

The config file specifies:
- `n_allvars=<int>` -- total variables passed (union of blocking and comparison)
- `n_comp=<int>` -- how many of those are comparison variables
- `comp_indices=<list>` -- which of the allvars are comparison variables (1-indexed)
- Blocking rules reference allvar indices
- Comparison specs reference allvar indices

The .ado builds the union of all variable names from `varlist` and blocking
rules, deduplicates, and maps indices.

### 3.3 Internal Data Structures

#### ComparisonSpec (replaces hardcoded N_LEVELS)

```c
typedef enum {
    CMP_JARO_WINKLER = 0,

    NULL_PENALIZE = 0,  /* null -> lowest level */
    NULL_NEUTRAL        /* null -> Bayes factor 1 (no contribution) */

typedef struct {

```

### 3.4 Comparison Vector Storage

In v1, comparison vectors are `int comp_vec[n_pairs * n_comp]` with a fixed
3 levels. In v2, each variable can have a different number of levels, but
comparison vectors remain integer-valued (level index). The EM and scoring code
index into per-variable `m[var][level]` and `u[var][level]` arrays.

```c
/* comp_vec layout: flat array, n_pairs * n_comp
 * comp_vec[pair_idx * n_comp + comp_idx] = level (0..n_levels[comp_idx]-1)
 * Level -1 = null (missing)
 */
int *comp_vec;
```

Using -1 for null allows the EM and scoring code to detect nulls and apply
neutral weighting when configured.

### 3.5 EM Changes for Variable-Length Levels

The EM arrays become ragged (each variable has its own n_levels):

```c
/* m_params[k] points to an array of n_levels[k] doubles */
double **m_params;  /* [n_comp][n_levels[k]] */
double **u_params;  /* [n_comp][n_levels[k]] */
```

Allocation:

```c
m_params = malloc(n_comp * sizeof(double *));
u_params = malloc(n_comp * sizeof(double *));
for (int k = 0; k < n_comp; k++) {
    int nl = config->comparisons[k].n_levels;
    m_params[k] = malloc(nl * sizeof(double));
    u_params[k] = malloc(nl * sizeof(double));
}
```

The EM E-step changes from:

```c
/* v1: fixed N_LEVELS */
int level = comp_vectors[i * n_comp + k];
double mk = m[k * N_LEVELS + level];
```

to:

```c
/* v2: variable levels */
int level = comp_vectors[i * n_comp + k];
if (level == -1) {
    /* null: skip this variable (neutral) or use lowest level (penalize) */
    if (config->comparisons[k].null_mode == NULL_NEUTRAL) {
        /* contribute nothing to log_m_prod or log_u_prod */
        continue;
    } else {
        level = config->comparisons[k].n_levels - 1;
    }
}
double mk = m_params[k][level];
double uk = u_params[k][level];
```

The M-step accumulation changes similarly -- skip null pairs in neutral mode,
or accumulate them in the lowest level in penalize mode.

For fixed m/u variables, the M-step skips updating those arrays.

### 3.6 Comparison Functions

New comparison functions added alongside the existing Jaro-Winkler:

```c
/* Returns a similarity score in [0, 1] or a distance >= 0 */
static double compare_jaro_winkler(const char *a, const char *b);     /* existing */
static int    compare_levenshtein(const char *a, const char *b);       /* edit distance */
static int    compare_damerau_levenshtein(const char *a, const char *b);
static double compare_jaccard_bigrams(const char *a, const char *b);   /* [0, 1] */
```

Level assignment depends on the method type:

- **Similarity methods** (JW, Jaccard): thresholds are similarity scores in
  descending order. `sim >= threshold[0]` -> level 0, `sim >= threshold[1]` ->
  level 1, etc.
- **Distance methods** (Levenshtein, Damerau-Levenshtein, numeric distance):
  thresholds are distance values in ascending order. `dist <= threshold[0]` ->
  level 0, `dist <= threshold[1]` -> level 1, etc.
- **Exact**: `a == b` -> level 0, else level 1.

```c
static int assign_level(CompSpec *spec, double score) {
    for (int t = 0; t < spec->n_levels - 1; t++) {
        if (spec->method <= CMP_JACCARD) {
            /* similarity: higher is better */
            if (score >= spec->thresholds[t]) return t;
        } else {
            /* distance: lower is better */
            if (score <= spec->thresholds[t]) return t;
        }
    }
    return spec->n_levels - 1;  /* lowest level */
}
```

### 3.7 Term Frequency Adjustment

TF adjustment modifies the match weight at scoring time (after EM). For each
pair where a TF-adjusted variable has an exact match (level 0), the
u-probability for that level is replaced with the term frequency of the matched
value.

The intuition: if two records both have `last_name == "Smith"`, the chance of a
random non-match having "Smith" is higher than for "Xylander". TF captures
this.

```c
/* At scoring time, for each pair i: */
for (int k = 0; k < n_comp; k++) {
    int level = comp_vec[i * n_comp + k];
    double mk = m_params[k][level];
    double uk = u_params[k][level];

    if (config->comparisons[k].tf_enabled && level == 0) {
        /* Replace uk with the term frequency of the matched value */
        char *value = get_string_value(k, pair_a[i]); /* or pair_b, same value */
        double tf = lookup_tf(config->tf_tables[k], value);
        if (tf > 0) uk = tf;
    }

    /* Bayes factor for this comparison */
    if (mk < 1e-10) mk = 1e-10;
    if (uk < 1e-10) uk = 1e-10;
    log_bf += log(mk) - log(uk);
}
```

The TF table is loaded from the config file into a hash table for O(1) lookup.
A simple open-addressing hash table is sufficient since table sizes are bounded
by the number of unique values.

**Important:** TF adjustment applies at scoring time, NOT during EM. The EM
estimates global m/u parameters. TF is a per-pair adjustment applied afterward
when computing final match weights and probabilities.

### 4.2 Multiple Blocking Rules -- Pair Generation with Deduplication

Each blocking rule generates candidate pairs independently. The union of all
pairs is the candidate set, deduplicated so each (i, j) pair appears once.

**Implementation strategy:** Process each rule sequentially. Use a hash set to
track which pairs have been seen.

```c
/* Pair hash set for deduplication across blocking rules */
typedef struct {
    int64_t *keys;    /* packed pair: (min(a,b) << 32) | max(a,b) */
    int      capacity;
    int      n_entries;
} PairSet;

static int64_t pair_key(int a, int b) {
    int lo = a < b ? a : b;
    int hi = a < b ? b : a;
    return ((int64_t)lo << 32) | (int64_t)hi;
}

/* For each blocking rule: */
for (int r = 0; r < n_rules; r++) {
    /* Build block keys for this rule (concatenate rule vars) */
    /* Sort observations by block key */
    /* Iterate blocks, generate pairs */
    for each pair (i, j) in block {
        if (has_link && same_source(i, j)) continue;
        int64_t key = pair_key(i, j);
        if (!pairset_contains(&seen, key)) {
            pairset_insert(&seen, key);
            /* Add to pair arrays */
            pair_a[n_pairs] = i;
            pair_b[n_pairs] = j;
            n_pairs++;
        }
    }
}
```

**Two-pass approach** (to avoid realloc):
1. First pass: count unique pairs across all rules (using hash set)
2. Allocate pair arrays
3. Second pass: populate pair arrays

Alternatively, use a growable array with doubling:

```c
/* Dynamic pair array */
int pair_cap = 1024;
int n_pairs = 0;
int *pair_a = malloc(pair_cap * sizeof(int));
int *pair_b = malloc(pair_cap * sizeof(int));

/* When adding: */
if (n_pairs >= pair_cap) {
    pair_cap *= 2;
    pair_a = realloc(pair_a, pair_cap * sizeof(int));
    pair_b = realloc(pair_b, pair_cap * sizeof(int));
}
```

The growable array approach is simpler and avoids the need for two full passes
through the blocking rules. Memory overhead of over-allocation is negligible
compared to the pair data itself.

### 4.3 TF Tables: .ado to Plugin

The .ado computes term frequencies and writes them into the config file:

```stata
* In .ado, for each TF-adjusted variable:
foreach v of local tfvars {
    * Compute frequency table
    tempvar tf_count tf_total tf_freq
    bysort `v': gen long `tf_count' = _N
    quietly count
    gen double `tf_freq' = `tf_count' / r(N)

    * Write to config file (unique values only)
    * ... write [TF_TABLE:<index>] section ...
}
```

The plugin loads TF tables into hash maps at config parse time.

### 4.4 Pairwise Output: Plugin to .ado

When `pairwise=1`, the plugin writes pairwise results to the output variables.
However, pairwise data has a different number of rows than the input data
(n_pairs rows vs n_obs rows), so it cannot be stored in the same dataset.

**Strategy:** The plugin writes pairwise results to a separate tempfile (CSV
or binary). The .ado reads it back.

```c
/* In the plugin, after scoring: */
if (config->pairwise) {
    FILE *pw = fopen(config->pairwise_path, "w");
    fprintf(pw, "_splink_id_a,_splink_id_b,_splink_match_weight,_splink_match_prob\n");
    for (int i = 0; i < n_pairs; i++) {
        double log_bf = compute_match_weight(i, ...);
        double prob = p_match[i];
        fprintf(pw, "%d,%d,%.8f,%.8f\n",
                pair_a[i] + 1,  /* 1-indexed for Stata */
                pair_b[i] + 1,
                log_bf, prob);
    }
    fclose(pw);
}
```

The .ado then:

```stata
* After plugin call, if pairwise requested:
if "`pairwise'" != "" {
    preserve
    import delimited "`pairwise_path'", clear
    save "`pairwise'", replace
    restore

    return local pairwise_file "`pairwise'"
}
```

The pairwise file path is communicated from .ado to plugin via the config file:

```
pairwise_path=<tempfile_path>
```

## 6. Backward Compatibility Guarantees

1. **Existing syntax works unchanged.** `splink varlist, block() gen()` with no
   `compare()` option uses v1 behavior exactly.

2. **Existing compiled plugins work.** The .ado detects whether the plugin
   supports v2 by attempting the v2 call. If it fails, falls back to v1 argv.
   (In practice, users will rebuild the plugin, but this provides a safety net.)

3. **Default thresholds match v1.** When `compare()` is not specified,
   strings get `jw(0.92, 0.80)` and numerics get `distance(0, 1.0)`.

4. **Default null behavior matches v1.** Without `nullweight(neutral)`,
   missing values are penalized (treated as lowest level), matching current
   behavior.

5. **r() results are a superset.** All existing r() results remain. New ones
   are added.

6. **Test suite must still pass.** All 14 existing tests remain valid for the
   default (v1-compatible) code path.
